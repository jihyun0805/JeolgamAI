# JeolgamAI 기능 요구사항 표 (상세)

| Domain | Actor | Priority | Title | Description | Preconditions |
|---|---|---|---|---|---|
| AUTH | System | P0 | 로그인 세션 검증 | 모든 보호 페이지(`dashboard`, `analysis`, `reports`, `execution-guide`, `integrations`)는 요청 시 서버에서 세션 유효성을 먼저 검증해야 하며, 세션이 없거나 만료된 경우 화면 데이터 렌더링 전에 로그인 페이지로 리다이렉트해야 한다. 클라이언트 캐시 데이터가 남아 있더라도 서버 응답 기준으로 접근을 차단해야 한다. | 사용자가 보호 URL로 직접 접근하거나 새로고침할 때 |
| AUTH | System | P0 | 어드민 접근 제어 | System Admin 전용 페이지는 `system_admin` 권한 클레임이 없는 사용자가 접근할 수 없어야 하며, 접근 시도는 감사 로그에 `forbidden`으로 기록되어야 한다. 권한 부족 사용자는 민감 데이터 일부라도 보지 못해야 하며, URL 추측 접근도 동일하게 차단되어야 한다. | System Admin 전용 라우트로 접근 시 |
| RBAC | System | P0 | 역할별 메뉴 가시성 제어 | 사이드바/상단 액션 버튼은 사용자 역할(System Admin, Company Admin, Company Operator)에 따라 노출 항목이 달라야 한다. 권한이 없는 기능 버튼은 비노출이 기본이며, 직접 API 호출 시도 시에도 서버 권한 검증에서 차단되어야 한다. | 로그인 완료 후 초기 레이아웃 렌더 시 |
| RBAC | Company Admin | P0 | 권고 승인 권한 | 권고 승인/반려 API는 Company Admin 역할만 호출 가능해야 하며, Operator가 요청하면 403을 반환해야 한다. 승인 성공 시 권고 상태 전이(Draft/Reviewed→Approved 또는 Rejected), 승인자 ID, 시각, 코멘트가 원자적으로 저장되어야 한다. | 분석 결과에 권고 항목이 생성된 상태 |
| RBAC | Company Operator | P1 | 실행 명령 조회 권한 | Company Operator는 실행/롤백 명령과 가이드를 조회할 수 있어야 하지만 승인 상태 변경은 수행할 수 없어야 한다. UI에서는 승인 버튼 비활성화, API에서는 403 반환으로 이중 통제해야 한다. | Operator가 권고 상세 또는 실행 가이드 화면을 열 때 |
| INTEGRATION | Company Admin | P0 | 연동 위저드 3단계 | 연동 설정은 입력→검증→샘플 분석 실행의 3단계 위저드로 제공해야 한다. 각 단계는 필수값 검증 후 다음 단계로 이동할 수 있으며, 실패 시 원인(권한 부족/네트워크/자격증명 오류)과 해결 가이드, 재시도 액션을 함께 보여줘야 한다. | Company Admin이 Integrations 화면에 진입했을 때 |
| INTEGRATION_AWS | Company Admin | P0 | AWS IAM Role 우선 연동 | AWS 연동은 Cross-account IAM Role 방식을 기본값으로 제공해야 하며, Role ARN/External ID 검증이 성공해야 저장할 수 있다. 검증은 `STS AssumeRole` 성공 여부와 필수 읽기 권한(Cost Explorer/CUR/EC2/RDS/S3 조회)을 포함해야 한다. | AWS 연동 추가 폼에서 인증 방식을 Role로 선택했을 때 |
| INTEGRATION_AWS | Company Admin | P1 | AWS Access Key 폴백 | Access Key 방식은 폴백으로 제공하며 키 입력 즉시 저장하지 않고 검증 후 시크릿 참조 형태로 저장해야 한다. 키 값은 로그/응답/화면에서 마스킹되어야 하고, 만료/비활성 상태 감지 시 사용자에게 갱신 알림을 표시해야 한다. | AWS 연동 폼에서 Access Key 방식 선택 시 |
| INTEGRATION_K8S | Company Admin | P0 | K8s 읽기 전용 토큰 검증 | Kubernetes 연동은 Read-only ServiceAccount 토큰을 사용해야 하며, 최소 조회 권한(노드/파드/네임스페이스/리소스 요청·한도)을 확인해야 한다. 검증 실패 시 누락된 RBAC 권한 목록을 메시지로 제공해야 한다. | K8s API 서버 URL과 토큰 입력 후 검증 버튼 클릭 시 |
| INTEGRATION_PROM | Company Admin | P0 | Prometheus API 검증 | Prometheus 연동은 베이스 URL과 토큰 유효성을 검사하고, 필수 쿼리(CPU/Memory usage, error rate, latency) 샘플 실행이 가능해야 성공으로 처리한다. 쿼리 일부 실패 시 부분 지원 상태로 저장하고, 미지원 항목을 명시해야 한다. | Prometheus 연동 정보 입력 후 검증 실행 시 |
| ANALYSIS | System | P0 | 수동 분석 실행 | 사용자가 분석 실행을 요청하면 최신 연동 상태를 기준으로 수집 작업을 시작하고 `analysisId`를 발급해야 한다. 진행 상태(queued/running/completed/failed)를 추적 가능해야 하며, 실패 시 부분 결과라도 제공 가능한 경우 부분 결과를 먼저 반환해야 한다. | 최소 1개 연동(AWS/K8s/Prom)이 활성 상태일 때 |
| ANALYSIS | Scheduler | P1 | 일 1회 배치 분석 | 시스템은 일 1회 자동 분석을 실행해야 하며, 실행 결과는 수동 분석과 동일한 스키마로 저장되어야 한다. 배치 실패 시 인앱 알림을 생성하고, 이전 성공 스냅샷을 대시보드 기준값으로 유지해야 한다. | 배치 스케줄 시각 도달 시 |
| ANALYSIS | System | P0 | 부분 연동 분석 허용 | AWS/K8s/Prom 중 일부만 연결된 경우에도 분석을 차단하지 않고 가능한 도메인 권고만 생성해야 한다. 단, 결과 화면과 점수 영역에 데이터 부족 경고 및 적용 범위를 명확히 표시해야 한다. | 1개 이상 연동 성공, 나머지 연동 실패/미설정 상태일 때 |
| SCORING | Recommendation Engine | P0 | 100점 스코어 산정 | 인프라 점수는 100점 만점으로 계산해야 하며 WA 6필러 80%, FinOps 1필러 20% 가중치를 적용한다. 각 필러 감점은 룰 기반으로 계산하고, 단일 룰 감점은 최대 8점으로 제한해 이상치 왜곡을 방지해야 한다. | 분석 스냅샷이 완료되어 권고 룰 평가 가능한 상태 |
| SCORING | Recommendation Engine | P0 | 등급 매핑 | 총점은 A/B/C/D/F 등급으로 매핑되어야 하며 컷은 A≥90, B≥80, C≥70, D≥60, F<60을 사용한다. 동일 점수대에서 최근 분석 대비 상승/하락 추세를 함께 표시해 개선 방향을 해석 가능하게 해야 한다. | 총점 계산 완료 후 대시보드/리포트 렌더 시 |
| SCORING | System | P0 | 점수 근거 전량 공개 | 점수 화면은 필러별 점수와 함께 룰별 감점 내역을 전부 보여줘야 하며, 각 감점 항목에 룰ID/원칙명/AWS 링크를 포함해야 한다. 사용자는 “왜 이 점수인지”를 추가 문의 없이 화면에서 확인할 수 있어야 한다. | 사용자 점수 상세 패널 진입 시 |
| SCORING | System | P1 | 점수 신뢰도 퍼센트 | 점수에는 데이터 품질 기반 신뢰도 퍼센트를 표시해야 하며, 연동 누락/메트릭 결손/관측 기간 부족 시 신뢰도를 낮춰야 한다. 신뢰도 계산식 요약을 툴팁으로 제공해야 한다. | 분석 결과를 대시보드 카드로 표시할 때 |
| RECOMMENDATION | Recommendation Engine | P0 | 권고 카드 표준 포맷 | 모든 권고는 최소 `근거`, `실행 명령`, `롤백 명령`, `월/연 절감 추정`, `리스크`, `신뢰도`, `출처` 필드를 가져야 한다. 필드 중 하나라도 계산 불가면 이유를 명시하고 빈 값으로 숨기지 않아야 한다. | 권고 생성 단계에서 도메인 룰 평가 완료 시 |
