# JeolgamAI 테스트 시나리오 (Given / When / Then)

## 문서 목적
- 기능 요구사항 문서(`functional-requirements-table.md`)를 테스트 가능한 시나리오로 전개한다.
- QA/개발/리뷰어가 동일한 기준으로 검증할 수 있도록 한다.

## 시나리오 목록

| ID | Domain | Priority | Requirement Title | Given | When | Then |
|---|---|---|---|---|---|---|
| TS-001 | AUTH | P0 | 로그인 세션 검증 | 보호 페이지 URL을 알고 있는 비로그인 사용자가 있다 | 사용자가 `/dashboard`를 직접 입력해 접근한다 | 서버는 페이지 데이터를 반환하지 않고 로그인 페이지로 리다이렉트한다 |
| TS-002 | AUTH | P0 | 로그인 세션 검증 | 만료된 세션 쿠키를 가진 사용자가 있다 | 사용자가 보호 페이지를 새로고침한다 | 서버는 세션 만료를 감지하고 401 또는 로그인 리다이렉트를 반환한다 |
| TS-003 | AUTH | P0 | 어드민 접근 제어 | `system_admin` 클레임이 없는 로그인 사용자가 있다 | 사용자가 System Admin 전용 URL에 접근한다 | 서버는 403을 반환하고 감사 로그에 `forbidden` 이벤트를 기록한다 |
| TS-004 | RBAC | P0 | 역할별 메뉴 가시성 제어 | Company Operator 계정이 로그인되어 있다 | 앱 레이아웃이 렌더링된다 | 승인/연동관리 메뉴는 비노출되고 조회/가이드 메뉴만 노출된다 |
| TS-005 | RBAC | P0 | 권고 승인 권한 | Company Operator 계정이 권고 승인 API endpoint를 알고 있다 | Operator가 승인 API를 호출한다 | 서버는 403을 반환하고 권고 상태를 변경하지 않는다 |
| TS-006 | RBAC | P0 | 권고 승인 권한 | Company Admin 계정과 `Reviewed` 상태 권고가 있다 | Admin이 승인 요청을 보낸다 | 권고 상태는 `Approved`로 전이되고 승인자/시각/메모가 저장된다 |
| TS-007 | INTEGRATION | P0 | 연동 위저드 3단계 | Company Admin이 Integrations 화면에 진입했다 | 1단계 필수 입력값 없이 "다음"을 클릭한다 | 시스템은 이동을 막고 누락 항목을 표시한다 |
| TS-008 | INTEGRATION | P0 | 연동 위저드 3단계 | 연동 검증 단계에서 네트워크 오류가 발생한다 | 사용자가 검증 버튼을 누른다 | 오류 원인/해결 가이드/재시도 버튼이 함께 표시된다 |
| TS-009 | INTEGRATION_AWS | P0 | AWS IAM Role 우선 연동 | AWS 연동 폼이 열려 있다 | 사용자가 폼을 처음 본다 | 인증 기본 선택값은 IAM Role 방식이다 |
| TS-010 | INTEGRATION_AWS | P0 | AWS IAM Role 우선 연동 | 잘못된 Role ARN이 입력되어 있다 | 사용자가 검증을 실행한다 | STS AssumeRole 검증이 실패하고 저장이 차단된다 |
| TS-011 | INTEGRATION_AWS | P1 | AWS Access Key 폴백 | Access Key 모드로 전환한 사용자가 있다 | 사용자가 키를 입력하고 저장한다 | 검증 성공 전에는 DB에 저장되지 않고 성공 후에도 평문 키는 저장되지 않는다 |
| TS-012 | INTEGRATION_K8S | P0 | K8s 읽기 전용 토큰 검증 | Read-only 권한이 일부 누락된 토큰이 있다 | 사용자가 K8s 검증을 실행한다 | 시스템은 누락된 RBAC 권한 목록을 반환한다 |
| TS-013 | INTEGRATION_PROM | P0 | Prometheus API 검증 | Prometheus 연결은 되지만 latency 쿼리가 실패한다 | 검증 실행이 완료된다 | 연동 상태는 `partial`로 저장되고 미지원 항목이 표시된다 |
| TS-014 | ANALYSIS | P0 | 수동 분석 실행 | 최소 1개 연동(AWS/K8s/Prom)이 활성이다 | 사용자가 "분석 실행" 버튼을 누른다 | `analysisId`가 즉시 발급되고 상태는 `queued`로 시작된다 |
| TS-015 | ANALYSIS | P0 | 부분 연동 분석 허용 | AWS만 성공, K8s/Prom은 미연동 상태이다 | 분석을 실행한다 | AWS 도메인 권고만 생성되고 결과 화면에 데이터 부족 경고가 표시된다 |
| TS-016 | ANALYSIS | P1 | 일 1회 배치 분석 | 배치 스케줄 시각이 도달했다 | 배치 잡이 실행된다 | 분석 스냅샷이 저장되고 실패 시 인앱 알림이 생성된다 |
| TS-017 | SCORING | P0 | 100점 스코어 산정 | 분석 스냅샷과 룰 평가 결과가 준비되어 있다 | 점수 계산 로직이 실행된다 | 총점은 0~100 범위이며 WA 80% + FinOps 20% 가중치가 적용된다 |
| TS-018 | SCORING | P0 | 100점 스코어 산정 | 특정 룰의 원시 감점값이 12점으로 계산됐다 | 감점 상한 규칙을 적용한다 | 해당 룰 최종 감점은 8점으로 제한된다 |
| TS-019 | SCORING | P0 | 등급 매핑 | 총점이 89점인 분석 결과가 있다 | 등급을 계산한다 | 결과 등급은 `B`가 된다 |
| TS-020 | SCORING | P0 | 등급 매핑 | 총점이 90점인 분석 결과가 있다 | 등급을 계산한다 | 결과 등급은 `A`가 된다 |
| TS-021 | SCORING | P0 | 점수 근거 전량 공개 | 점수 상세 화면을 연 사용자가 있다 | 감점 내역을 확인한다 | 모든 감점 항목에 ruleId/principle/awsLink가 표시된다 |
